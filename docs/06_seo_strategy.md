# SEO Strategy

## SEO goal

The website should be launchable as a serious multilingual B2B catalog, with technical SEO handled as part of the product foundation rather than added later.

The main SEO assets are:
- home page
- category pages
- product pages
- manufacturer pages
- about page

## URL strategy

### Route format

Use locale-prefixed public routes only:

- `/en`
- `/en/about`
- `/en/products`
- `/en/categories`
- `/en/categories/{...categoryPath}`
- `/en/products/{slug}`
- `/en/manufacturers/{slug}`
- `/en/contact`
- mirrored under `/fr`

Root behavior:
- `/` redirects to `/en`
- root itself is not indexed

### Why this is the right strategy

- clean separation between languages
- easier canonical logic
- easier hreflang clusters
- simpler sitemap generation
- no ambiguity about which locale a crawler is seeing

## Slug rules

- Slugs are locale-specific
- Slugs are stored in translation rows, not inferred on the fly
- Slugs must be lower-case, URL-safe, and kebab-case
- Slugs should not include random IDs in public URLs
- Slug uniqueness must be enforced per content type and locale
- Category pages use localized hierarchy paths rather than flat single-segment slugs
- Slug changes after publication should create redirects during implementation

Examples:
- `/en/products/citric-acid-monohydrate`
- `/fr/products/acide-citrique-monohydrate`
- `/en/categories/food-additives/acidity-regulators`
- `/fr/categories/additifs-alimentaires/regulateurs-d-acidite`

## Metadata rules

Every public page should support at least:

- `title`
- `meta description`
- canonical URL
- Open Graph title
- Open Graph description
- Open Graph image fallback

Entity-specific rules:

### Home

- strongest title and description writing effort
- must describe Abadis Tejarat Arka and its trading focus clearly

### Category pages

- metadata should describe the category branch and catalog relevance
- avoid boilerplate duplication across parent and child categories

### Product pages

- metadata should include product name and company context
- avoid thin autogenerated descriptions
- if the page shows no public price, metadata should still emphasize inquiry intent rather than imply ecommerce

### Manufacturer pages

- metadata should explain the manufacturer and its relationship to ATA's offering

## Canonical strategy

- Each public localized page gets a self-referencing canonical
- Canonicals point to the final locale-specific public URL
- No canonical should point from French content to English content
- Draft/admin/preview routes must not emit indexable canonicals

Examples:
- `/en/products/citric-acid-monohydrate` canonicals to itself
- `/fr/products/acide-citrique-monohydrate` canonicals to itself

## Hreflang strategy

For every entity with multiple public locales:
- emit `hreflang="en"`
- emit `hreflang="fr"`
- emit `hreflang="x-default"` pointing to `/en/...`

Do not emit `fa` while Persian is not public.

Important rules:
- only include locale alternatives that are actually published and publicly routable
- do not include broken or draft alternates just because the base entity exists

## Robots strategy

### Public pages

- allow crawling/indexing of published public pages

### Block or noindex these areas

- `/admin`
- `/api`
- preview routes
- draft-only routes if any exist
- root redirect endpoint if implemented as a route

### Persian before launch

If `/fa/...` routes exist internally during implementation:
- block them from indexing
- preferably avoid exposing them publicly at all until enabled

## Sitemap strategy

Recommended output:
- one sitemap index or one unified sitemap generated by the app
- include only published, public, locale-valid URLs

Include:
- `/en`
- `/fr`
- published page URLs
- published category URLs
- published product URLs
- published manufacturer URLs
- public product document URLs only if they are intentionally indexable

Exclude:
- draft content
- archived content
- admin routes
- root redirect
- non-public Persian URLs
- non-public product documents

## Heading and semantic HTML expectations

Every public page should follow a predictable semantic structure:

- exactly one meaningful `h1`
- logical `h2` and `h3` hierarchy
- list semantics for product/category/manufacturer listings
- `<nav>` for main nav and breadcrumbs
- `<main>` for primary content
- `<footer>` for site footer
- accessible labels for forms and interactive filters

This is both an accessibility and SEO requirement.

## Internal linking strategy

The site should deliberately cross-link across the three catalog entity types.

Required internal link flows:

- home -> featured categories
- home -> featured products
- home -> featured manufacturers
- category -> child categories when applicable
- category -> products
- product -> category
- product -> manufacturer
- manufacturer -> products
- about/contact -> products and inquiry paths where relevant

Each detail page should also include breadcrumbs:

- Home
- Section index
- Parent category segments where relevant
- Current entity

## Image and document SEO requirements

### Product images

- every product should have at least one primary image before publication
- alt text should be based on localized product name, not a file name
- decorative imagery should use empty alt only when truly decorative

### Manufacturer images/logos

- use manufacturer name in alt text when the asset conveys identity

### Product documents

- public documents should use descriptive file labels, not opaque filenames
- if a document is public and useful for search, its landing link context should be indexable even if the raw file itself is not emphasized
- avoid exposing empty or duplicate files just to create more indexable URLs

### CMS validation expectation

- media uploads should require title/usage clarity
- product forms should warn when no publishable image exists
- public document entries should warn when no localized label exists

## Pricing SEO guidance

- Product pages may optionally show a public price
- If no public price is set, the page must show a localized contact-for-pricing message
- Pages without a public price must not imply online purchase capability
- Structured data should not emit ecommerce offers by default
- If public prices are later exposed in structured data, they must remain informational and not suggest checkout or online payment

## Indexability rules

Public indexing is allowed only when:

- base entity is `published`
- locale translation is `published`
- locale is public
- required slug/path exists
- required metadata exists
- minimum content threshold exists

Anything else should be:
- absent from sitemap
- absent from hreflang
- rendered as non-public or 404

## Minimum content requirements before publication

### Home

- hero heading and subheading
- at least one trust block
- at least one CTA
- featured content configured
- metadata completed

### Category page

- localized name
- valid localized slug path
- short description
- meaningful intro/body
- at least one published product in that locale
- metadata completed

### Product page

- localized product name
- slug
- short description
- long description or structured equivalent
- availability status and message
- manufacturer linked
- category linked
- at least one image
- metadata completed
- if no public price is set, contact-for-pricing copy exists

### Manufacturer page

- localized manufacturer name
- slug
- summary
- body
- at least one related published product
- metadata completed

## Structured data opportunities

### MVP-safe structured data

- `Organization` on home/about
- `WebSite`
- `BreadcrumbList`
- `ItemList` for category/manufacturer/product indexes where appropriate
- `Product` on product pages only if kept informational

### Conditional structured data

- `Offer` only if a public price is intentionally displayed and the implementation avoids implying checkout, shipping, or online transaction support
- `FAQPage` only when a real visible FAQ section exists on the public page

## Future SEO expansion opportunities

- optional FAQ content blocks on category/product/manufacturer pages
- optional downloads hub when datasheets and catalogs are consistently available
- optional industries/applications landing pages when content exists
- optional blog/content hub only if ATA commits to sustained editorial production

## Search-console and measurement workflow

Recommended launch workflow:

1. connect Google Search Console for the production domain
2. submit sitemap
3. verify locale alternates and canonicals
4. monitor crawl/index coverage
5. review top landing pages and underperforming category/product pages

Operational workflow:
- use Search Console and simple traffic logs to identify which product/category/manufacturer pages attract search traffic
- improve thin but promising pages before adding large new sections
- treat paid analytics and paid SEO tooling as optional later upgrades, not MVP dependencies

## SEO conclusion

The MVP should optimize for a small number of strong, well-structured, multilingual landing surfaces rather than a large volume of weak pages.

That means:
- strict publication gates
- localized slugs and hierarchy paths
- clean canonicals
- accurate hreflang
- content quality before indexation
